# Ouverture — Spec

Ouverture is a deployable web app built with Next.js (App Router) that helps developers plan and execute Claude Code work. Users sign up, log in, and manage their own projects, tasks, phases, and tickets. The AI backbone is MiniMax, behind an adapter interface for future swapping. Data is stored in SQLite via Prisma. The UI is dark theme only.

---

## Data model

The hierarchy is: **Project → Task → Phase → Ticket**

Every level has a `status` field of `active | done`. Marking a parent done cascades to all children. Nothing is deleted on mark done — done items move to a collapsed done section within their parent view and can be re-opened individually. Re-opening a parent does not cascade to its children.

### User
- `id` — uuid
- `email` — string, unique
- `passwordHash` — string
- `minimaxApiKey` — string, encrypted at rest
- `createdAt` — ISO timestamp

### Project
- `id` — uuid
- `userId` — references User
- `name` — string
- `spec` — free-text description of what the project is about
- `createdAt` / `updatedAt` — ISO timestamps

### ProjectTool
- `id` — uuid
- `projectId` — references Project
- `type` — agent | skill | mcp
- `name` — string

### Task
- `id` — uuid
- `projectId` — references Project
- `userId` — references User
- `title` — string
- `requirements` — structured requirements text generated by AI on finalize
- `conversationHistory` — JSON, full chat history from requirements gathering
- `status` — active | done
- `createdAt` / `updatedAt` — ISO timestamps

### Phase
- `id` — uuid
- `taskId` — references Task
- `title` — string
- `description` — string
- `estimateHours` — string, e.g. "3–5h"
- `priority` — high | medium | low
- `status` — active | done

### PhaseTooling
- `id` — uuid
- `phaseId` — references Phase
- `type` — agent | skill | mcp
- `name` — string
- `isNew` — boolean, true if AI suggested this as new tooling not in the project
- `rationale` — string, optional, AI explanation for why this tool is needed

### Ticket
- `id` — uuid
- `phaseId` — references Phase
- `title` — string
- `description` — string
- `instructions` — JSON, string[], step-by-step instructions
- `claudeCodePrompt` — string, ready-to-paste Claude Code terminal prompt
- `testPrompt` — string, prompt to verify acceptance criteria and check for regressions
- `acceptanceCriteria` — JSON, string[], verifiable conditions
- `status` — active | done
- `order` — integer

### TicketTooling
- `id` — uuid
- `ticketId` — references Ticket
- `type` — agent | skill | mcp
- `name` — string
- `isNew` — boolean
- `rationale` — string, optional

## Features

### Auth

The app requires authentication. Users sign up with email and password. All data is scoped to the authenticated user — no user can access another user's data. Auth is handled with NextAuth.js v5. Passwords are hashed with bcrypt. Sessions use JWT stored in HTTP-only cookies.

Every route except `/login` and `/signup` requires an authenticated session. Unauthenticated requests to API routes return 401.

### Projects

The app opens to a projects page listing all projects belonging to the authenticated user as cards. Each card shows the project name and spec summary. Projects can be created, edited, and deleted.

The project inner page shows the project name, full spec, all tasks associated with the project, and the project's registered agents, skills, and MCPs. Agents, skills, and MCPs are entered manually by the user — there is no filesystem scanning. The user can add, edit, and remove tooling from a project at any time.

### Tasks

A task is created from within a project. Creating a task opens a chat interface. The user describes what they want to build and the AI (requirements analyst role) responds with clarifying questions. The number of clarifying questions adapts to the complexity of the task — a simple task may need only one or two rounds, a complex task may go through several. This continues until the user clicks Finalize.

During the conversation the AI may suggest new agents, skills, or MCPs inline in the chat if it identifies a gap that the project's existing tooling does not cover.

On finalize, the full conversation is sent to MiniMax which returns structured task requirements as JSON. The task is saved and listed under the project. Tasks can be viewed, edited, and deleted.

### Phases

From a task, the user can generate phases. Ouverture sends the task requirements plus the project's registered tooling to MiniMax (implementation architect role). MiniMax returns a list of phases each with a title, description, hour estimate, suggested tooling, and optional new tooling suggestions.

The user reviews the proposed phases, adjusts the tooling selection per phase, and confirms. New tooling suggestions are shown with a badge and rationale. Phases can be viewed, edited, and deleted after confirmation.

### Tickets

From a phase, the user can generate tickets. MiniMax (ticket engineer role) returns a list of tickets based on the phase details and task requirements. Each ticket contains a title, description, step-by-step instructions, a ready-to-paste Claude Code terminal prompt, acceptance criteria, a test prompt, tooling references, and optional new tooling suggestions.

The user reviews the proposed tickets, edits any field, adds or removes tickets, and confirms. Tickets can be viewed, edited, and deleted after confirmation. The Claude Code prompt and test prompt each have a one-click copy-to-clipboard button. New tooling suggestions are shown with a badge and rationale.

### Done system

| Action | Effect |
|---|---|
| Mark ticket done | Ticket moves to done section within its phase |
| Mark phase done | Phase + all its tickets → done |
| Mark task done | Task + all phases + all tickets → done |

Done sections are collapsed by default and expandable within their parent view. Any done item can be re-opened individually. Re-opening a parent does not cascade to its children.

## AI roles

All AI calls go through Next.js API routes. The MiniMax integration is built behind an `AIAdapter` interface so alternative models can be swapped in the future without changing any calling code. The authenticated user's MiniMax API key is retrieved from the database on each request — it is never sent to the client or stored in environment variables.

### Requirements analyst
Used during task creation chat. Streams responses via SSE. Adapts the number of clarifying questions to the complexity of what the user describes. May suggest new agents, skills, or MCPs inline when it identifies gaps in the project's existing tooling. On finalize, returns structured task requirements as JSON.

### Implementation architect
Called on phase generation. Receives task requirements and the project's registered tooling. Returns a JSON array of phases with title, description, hour estimate, tooling selections, and new tooling suggestions.

### Ticket engineer
Called on ticket generation. Receives phase details and task requirements. Returns a JSON array of tickets with all fields populated.

## Technical constraints

- Next.js 16 App Router
- Dark theme only — no light mode, no toggle
- NextAuth.js v5 for authentication
- Prisma + SQLite for data storage
- MiniMax API key stored per user in the database, encrypted at rest
- All MiniMax calls are server-side only — the API key is never sent to the client
- AI integration is behind an `AIAdapter` interface — MiniMax is the default implementation
- Deployable to any Node.js host (Railway, Render, VPS)
